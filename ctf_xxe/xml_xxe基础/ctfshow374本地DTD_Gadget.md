# ctfshow374：本地 DTD Gadget

这篇是三种方法里最绕的一篇，但它解决的是一个很真实的问题：

> 有时候你知道要用参数实体、要做复杂拼接，可是直接在自己写的那段 XML 里不好写，或者会被 DTD 规则卡住。

这时就会想到一个进阶技巧：

- 不完全依赖你自己写的 DTD
- 改去“借用服务器本地已经存在的 DTD 文件”

这就是常说的 **本地 DTD Gadget**。

---

## 1. 先别急着记 payload，先理解“为什么会有这条路”

在 [[ctfshow374OOB_HTTP外带]] 里，我们用了远程 DTD：

- 目标服务器先来你这里下载 `evil.dtd`
- 再按里面的规则执行

这种方法很好理解，但现实里可能遇到这些问题：

- 目标出网受限
- 你不想依赖外部网络
- 某些参数实体写法在当前上下文里受限

于是就有人想到：

“服务器自己机器上，不是经常本来就装着一堆 `.dtd` 文件吗？”

比如某些 Linux 组件、文档工具、桌面程序，会自带 DTD 文件。

如果 XML 解析器允许读取本地文件，那么这些 DTD 文件就可能被拿来当“跳板”。

---

## 2. 什么叫 Gadget

这里的 `Gadget`，你可以先粗暴理解成：

> 一个可以借来用的现成零件。

本地 DTD Gadget 的意思就是：

- 系统上已经有一个 DTD 文件
- 这个 DTD 文件里已经定义了一些参数实体
- 你可以想办法“改写”其中某个参数实体
- 然后借它自己的展开流程，帮你把恶意内容拼出来

它像不像“借壳执行”？很像。

---

## 3. 常见示例

很多文章会举类似这样的 payload：

```xml
<!DOCTYPE foo [
  <!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">
  <!ENTITY % ISOamsa '
    <!ENTITY %% file SYSTEM "file:///flag">
    <!ENTITY %% oob "<!ENTITY exfil SYSTEM ''file:///%%file;''>">
    %%oob;
  '>
  %local_dtd;
]>
```

你第一次看到它很容易头皮发麻，这是正常的。它的阅读方式不该是“逐字符硬背”，而应该拆成三层。

---

## 4. 第一层：先把本地 DTD 拉进来

```xml
<!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">
```

这句是在说：

- 去服务器本地读一个现成的 DTD 文件

也就是把系统自带的规则文件载入当前 XML 解析流程。

---

## 5. 第二层：提前重定义它会用到的某个参数实体

```xml
<!ENTITY % ISOamsa ' ... '>
```

这里的关键不是 `ISOamsa` 这个名字本身，而是：

- 目标 DTD 里本来就会用到这个参数实体
- 你提前把它重新定义成你自己的内容

等本地 DTD 后面展开 `%ISOamsa;` 时，它展开的就不再是原来的安全内容，而是你塞进去的恶意内容。

这就是“重定义”的核心。

---

## 6. 第三层：在重定义内容里塞真正的利用逻辑

重定义块里通常会做这些事：

- 读本地文件
- 再构造另一个实体
- 最后触发某种外带或报错

比如示例里你能认出两件事：

```xml
<!ENTITY %% file SYSTEM "file:///flag">
```

这一步是在读 `/flag`。

然后：

```xml
<!ENTITY %% oob "<!ENTITY exfil SYSTEM ''file:///%%file;''>">
```

这一步是在把刚才读到的内容再塞进另一个实体定义里。

最后：

```xml
%%oob;
```

让它真正展开。

---

## 7. 为什么这能绕过一些限制

这是这篇最重要的一点。

很多复杂 XXE payload 之所以难写，不是因为“思路想不到”，而是因为 XML 对 DTD 有不少语法限制。

尤其是：

- 参数实体只能在某些位置用
- 某些嵌套写法在内部子集里不稳定
- 直接写复杂声明时，解析器可能报语法错误

而本地 DTD Gadget 的思路是：

- 让一个“外部 DTD 文件”替你完成后续展开
- 你只负责提前把某个关键参数实体换掉

于是很多原本在内部子集不好完成的操作，就借外部 DTD 的执行流程做出来了。

你可以把它理解成：

> 不是我自己直接把整条链写出来，而是我偷偷修改了某个中间零件，让系统替我把后半段跑完。

---

## 8. 这条路线常和什么配合

本地 DTD Gadget 自己通常不是最终目的，它更像“搭桥方法”。

最终它常常会配合：

- [[ctfshow374报错外带]]
- [[ctfshow374OOB_HTTP外带]]

也就是说：

- Gadget 负责帮你把复杂实体链拼起来
- 报错或 HTTP 请求负责把数据真正带出来

---

## 9. 为什么它比前两种更进阶

因为它要求你同时理解三件事：

1. XML / DTD / 实体的基本语法
2. Blind XXE 为什么需要“间接拼装”
3. 系统自带 DTD 的实体展开顺序

所以如果你现在觉得这篇比前两篇难很多，不是你有问题，而是它确实就是更进阶的话题。

建议顺序一定是：

1. [[xml基础]]
2. [[ctfshow374OOB_HTTP外带]]
3. [[ctfshow374报错外带]]
4. [[ctfshow374本地DTD_Gadget]]

---

## 10. 使用这条方法时的现实问题

### 1. 路径不固定

本地 DTD 文件路径和系统环境强相关。

例如：

- 某台机器有 `/usr/share/yelp/dtd/docbookx.dtd`
- 另一台可能根本没有

所以这类 payload 经常需要按环境调整。

---

### 2. 实体名也不固定

你不是随便找个名字重定义就行，而是要找：

- 本地 DTD 里真的存在
- 后续真的会被引用

的那个参数实体。

---

### 3. 调试成本高

这类 payload 非常容易因为：

- 引号
- 百分号
- 分号
- 嵌套层级

而失败。

所以它通常不是最先尝试的路线，而是“你已经懂 Blind XXE 了，开始处理更复杂约束”时才会用。

---

## 11. 一句话总结

本地 DTD Gadget 的本质是：

> 借服务器现成的 DTD 文件当跳板，通过重定义其中的参数实体，把原本不好直接写出来的 XXE 链条拼出来。

如果你把这句话吃透，再回头看这题，就会明白它并不是一条“完全不同”的方法，而是对 Blind XXE 利用链的一种进阶组装技巧。
