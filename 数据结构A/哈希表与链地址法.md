# 哈希表与链地址法

## 1. 核心痛点：为什么要用哈希表？

在传统的数据结构中，如果我们要寻找一个特定名字对应的数据（例如找“张三”的成绩），通常只能进行**顺序查找**，时间复杂度为 $O(N)$，非常慢。
而**哈希表（散列表）**的出现，是为了实现**“瞬间拿取”**（时间复杂度 $O(1)$）。它通过计算，直接算出存放位置。

## 2. 魔法盒子：散列函数

如何把文字（如“张三”）变成数组的数字下标？我们需要一个**散列函数（Hash Function）**。
- **作用：** 将复杂的键（Key，如名字）通过数学计算，转换成一个固定的数字（即数组的索引下标）。
- **流程：** 输入“张三” -> 散列函数计算 -> 输出下标 `5` -> 直接去数组 `[5]` 的位置读取数据。

## 3. 不可避免的麻烦：冲突（Collision）

由于数组空间有限，而可能的输入是无限的，不同的输入有可能会计算出相同的数字下标。
例如：“张三”计算出 `5`，“李四”经过散列函数计算，结果也碰巧是 `5`。这就叫做**冲突**。

## 4. 解决冲突的最经典招式：链地址法（拉链法）

当发生冲突时，我们不能把后面来的数据丢掉，也不能硬塞进同一个格子里。**链地址法**的思路是：在发生冲突的数组格子里，挂一条**链表**。

### 直观模型
可以把哈希表想象成存包柜，如果几个人算出来的柜号都是 5，就在 5 号柜门上垂下一条绳子，大家依次把自己挂在绳子上。

```text
哈希表（数组下标）
[ 0 ] -> 空闲
[ 1 ] -> [ 钱七的档案 ] -> NULL
[ 2 ] -> 空闲
[ 3 ] -> [ 赵六的档案 ] -> NULL
[ 4 ] -> 空闲
[ 5 ] -> [ 李四的档案 ] -> [ 张三的档案 ] -> NULL   <-- 发生冲突，用链表串起来
[ 6 ] -> 空闲
```

## 5. 链地址法的重要特性（考点提炼）

针对链地址法的常见考点，我们总结如下：

- **插入时间是相同的（若采用头插法）：** 不管链表多长，如果总是往链表最前面插，只需执行固定次数的操作（$O(1)$ 时间复杂度）。
- **不会引起二次聚集现象：** 冲突的数据只会在自己的抽屉下排队，**绝不占用其他抽屉的空间**。这与“开放定址法”不同，不会导致连续拥堵。
- **查找时间是不相同的：** 在链表中查找元素需要从头遍历。排在前面的元素瞬间就能找到，排在后面的需要找很久，时间显然不同。
- **适合表长不确定的情况：** 链表的特性是动态分配内存，只要内存够，冲突再多也能挂得下，不会出现“表满了完全塞不进去”的死胡同。
