# 御网杯线下赛大模型局域网部署方案

## 背景与痛点
在CTF线下赛（如御网杯）中，团队需要本地的大模型算力支持（如代码审计、流量分析解密、自动化Pwn脚本编写）。但每个人单机算力有限，官方提供的网络基础设施（路由器）带宽和延迟往往不佳。

直接在官方路由器下使用 [llama.cpp](https://en.wikipedia.org/wiki/Llama.cpp) 的 RPC 进行分布式推理（流水线并行），会因为极低的带宽和高延迟导致模型生成速度卡死（传输激活张量耗时远大于计算耗时）。

## 解决方案演进

### 1. 任务级解耦与请求路由（最稳妥）
- **核心思想**：放弃切分大模型，改为“多机部署多个小模型”。
- **架构**：每个队员笔记本运行一个量化版的小模型（如 Qwen2.5-7B-Instruct Q4_K_M）。
- **工具**：使用 [LiteLLM](https://github.com/BerriAI/litellm) 或 Nginx 搭建 API 网关。
- **优势**：网络层只传输纯文本（Prompt），即使官方路由器带宽只有 10Mbps 也能毫无延迟地工作。

### 2. 物理网络劫持与自建高速子网（实干家方案）
- **核心思想**：绕过官方劣质路由器，搭建团队专属的物理高速通道。
- **架构**：
  - 团队自带一个 **2.5G 或 10G 的交换机**。
  - 所有队员通过超六类（CAT6）网线连接到该交换机。
  - 选出一台机器作为 **NAT网关**（或串联一个便携路由器），一头连己方交换机，另一头连官方路由器。
- **优势**：在自建的 2.5G 局域网内，完全可以支撑 `llama.cpp` 的 RPC 分布式推理，跑起 32B 甚至 70B 的大模型。

### 3. 异构算力下的非对称流水线切分（高阶踩坑）
在自建高速子网的前提下，如何高效利用不同配置的笔记本（如 Mac M1 + 游戏本 4060 + 轻薄本 CPU）：
- **非对称层切分**：将模型层级按机器算力（FLOPs）进行比例分配（`--tensor-split` 或手动指定 rpc server）。
- **避坑指南**：
  - 必须关闭 Windows Defender 等防火墙，否则 RPC 端口（默认 50052）极易被拦截或降速。
  - 第一层（Token Embedding）和最后一层（Logits 输出）一定要放在算力最强、网络最核心的机器上，减少最终答案汇聚的延迟。

## 推荐技术栈
- **[Exo](https://github.com/exo-explore/exo)**: 专为日常消费级设备（手机、Mac、PC）设计的异构集群 AI 推理框架，对弱网环境容忍度比纯 RPC 更高。
- **[Petals](https://github.com/bigscience-workshop/petals)**: 类 BT 种子网络的大模型分布式框架。
