# 防御修复与审计

## 修复原则（从强到弱）
1. 业务去 SSRF 化：不要让用户直接控制服务端请求目标。
2. 严格白名单：只允许固定域名/固定路径/固定协议。
3. 解析后校验：对最终解析 IP 做网段限制（拒绝内网、回环、链路本地）。
4. 禁止自动重定向或每跳都做同样校验。
5. 出网控制：防火墙/安全组限制实例访问敏感地址。

## 不能只做的事
- 仅黑名单关键字（`localhost`、`127.0.0.1`）。
- 仅正则匹配 URL 字符串，不做 DNS 解析后检查。
- 仅应用层修补，不做网络层隔离。

## 参考安全实现要点
- 协议白名单：仅 `http`/`https`。
- host 白名单：固定业务域名。
- DNS 解析得到全部 A/AAAA 记录后校验。
- 再请求前和重定向后都校验目标 IP。
- 请求库关闭危险协议 wrapper。

## 审计清单
- 代码中搜索：`requests.get(url)`、`curl_exec($url)`、`open-uri`、`axios(url)`（服务端场景）。
- 关注所有“代请求”功能：抓图、下载、回调、预览。
- 检查是否存在可控重定向链。

## 蓝队加固建议
- 对 `169.254.169.254` 做网络层阻断（按业务例外放通）。
- 元数据服务启用更严格版本（如 token 机制）。
- 异常出网行为告警（目标 IP、端口、频率）。
