# 防御修复与审计

## 最重要的原则
- 永远不要把用户输入拼接成模板源码再渲染。
- 用户输入应该作为数据传给固定模板，而不是作为模板语法的一部分。

## 错误示例
```python
return render_template_string("Hello " + user_input)
```

## 正确方向
```python
return render_template("hello.html", name=user_input)
```

模板固定，用户输入只是变量值。

## 具体修复建议

### 建议 1：禁用动态模板拼接
- 审计所有 `render_template_string`、动态模板解析接口、字符串模板资源。
- 对“预览模板”“测试模板”功能做重点排查。

### 建议 2：用户可配模板必须走白名单能力
- 只允许占位符，不允许表达式、函数、对象访问。
- 例如只允许 `${name}` 这类简单变量替换，后端自行实现有限替换器。

### 建议 3：减少模板上下文暴露
- 不把请求对象、应用对象、配置对象无必要地塞进上下文。
- 少一个对象，就少一条可能的利用链。

### 建议 4：开启沙箱不等于结束
- 沙箱只是提高攻击成本。
- 仍要做源头修复，不能把安全寄托在黑名单上。

### 建议 5：对模板错误统一处理
- 不向前端暴露详细栈追踪、模板路径、类名。
- 报错脱敏可以明显降低攻击者识别效率。

## 代码审计重点
- 搜索关键危险点：

  - 动态加载模板字符串的接口
- 搜索业务关键字：

  - message format
  - custom theme

## 黑名单为什么不靠谱
- 拦截 `{{` 不能解决 `${}`、`#{}`、`<% %>` 等其它语法
- 拦截 `class` 不等于拦截所有对象访问链
- 过滤规则很难覆盖不同模板引擎和所有等价写法

## 开发团队应该有的认知
- 模板不是普通字符串。
- 模板引擎是解释器，给它喂用户输入就等于把攻击面交出去。
