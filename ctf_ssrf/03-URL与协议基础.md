# URL 与协议基础（打 SSRF 前必须会）

## URL 基本结构
`scheme://userinfo@host:port/path?query#fragment`

CTF 中重点字段：
- `scheme`：协议（http、https、file、gopher、dict）
- `host`：目标主机（IP/域名/IPv6）
- `port`：目标端口
- `path`：目标资源路径

## 常见协议在 SSRF 中的意义
- `http/https`：基础探测、请求内网 Web。
- `file://`：尝试本地文件读取（是否可用取决于后端库和 wrapper）。
- `gopher://`：可构造原始 TCP payload（进阶重点，见 [[10-gopher-redis-mysql利用]]）。
- `dict://`：某些环境可用于探测或命令拼接（可用性不稳定）。

## 特殊主机写法（绕过基础）
- `127.0.0.1`
- `localhost`
- `0.0.0.0`
- `2130706433`（十进制 IP）
- `017700000001`（八进制表示）
- `0x7f000001`（十六进制表示）
- `[::1]`（IPv6 loopback）

## URL 解析差异点
- `@`：`http://a.com@127.0.0.1/` 实际 host 是 `127.0.0.1`。
- `#`：片段通常不发给服务端，但可能影响前置校验。
- `%xx` 编码：多次解码可能触发绕过。
- 重定向：先过白名单后跳转到内网是经典问题。

## 初学者必做实验
1. 对同一后端传入 6 种 localhost 写法，记录哪些被拦截。
2. 测试是否跟随 302 重定向。
3. 测试协议白名单是校验“前缀字符串”还是“真实解析结果”。
