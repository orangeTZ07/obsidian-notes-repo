# 漏洞成因与利用条件

## 漏洞成因分类

### 1. 业务功能天然需要“代请求”
- 图片抓取（avatar from URL）
- 网页快照（screenshot）
- 文档导入（read from url）
- webhook / callback 验证

这类功能不是错，错在“直接信任用户给的目标地址”。

### 2. 防护逻辑只做了字符串过滤
- 只禁 `127.0.0.1` 但放过 `2130706433`
- 只禁 `http://` 却未限制 `gopher://`
- 只看第一次解析结果，没处理重定向

### 3. 信任边界错误
- 认为“请求从本机发起就安全”
- 认为“内网服务不需要鉴权”

## SSRF 成功利用三要素
- 可控输入：能改 host / port / path / scheme 中至少一个。
- 可达网络：应用服务器能访问目标。
- 可观测反馈：有响应、时间差、日志或外带信号。

## 输出通道类型
- 直显型：响应内容直接回传前端。
- 半盲型：只回状态码/长度/报错。
- 全盲型：无回显，靠 DNS/HTTP 外带与时延判断。

## 做题思考模板
1. 输入点在哪里？
2. 我控制到 URL 的哪一部分？
3. 后端使用哪种请求库？是否跟随重定向？
4. 哪些协议可用？
5. 有没有防护（黑名单/白名单/正则）？
6. 反馈通道是什么？
7. 内网优先探测哪些目标？
