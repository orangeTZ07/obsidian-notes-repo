# 识别与打点方法

## 第一步：判断哪里可能发生模板渲染
- 用户输入后被“格式化展示”的地方优先。
- 特别关注：
  - 预览页
  - 欢迎页
  - 邮件内容页
  - 后台配置的模板片段
  - 错误消息中的动态字符串

## 第二步：丢最小探测 payload

### 通用基础集
```txt
{{7*7}}
${7*7}
#{7*7}
<%= 7*7 %>
{{7+7}}
{{1337}}
```

### 观察点
- 是否变成 `49`
- 是否原样输出
- 是否被 HTML 编码
- 是否触发报错
- 响应长度是否变化

## 第三步：区分三种情况

### 情况 1：原样输出
- 可能没有进模板解析器。
- 也可能进了模板，但语法不是当前这套。
- 下一步应该切换分隔符和引擎语法继续试。

### 情况 2：报错
- 价值很高。
- 报错里可能直接带出引擎名、模板文件路径、行号、对象名。
- 先保存原始错误页面，后面做复盘用。

### 情况 3：表达式成功求值
- 立刻进入指纹识别和对象探测阶段。
- 不要浪费时间继续刷简单数学题。

## 第四步：探上下文变量

### Jinja2 常见起点
```jinja2
{{ config }}
{{ request }}
{{ self }}
{{ url_for }}
{{ get_flashed_messages }}
```

### 为什么这些对象重要
- `request`：常能碰到应用上下文。
- `config`：可能直接泄露密钥或配置路径。
- `self`：模板内部对象，常用于继续扩展。
- 全局函数：可能带到 `__globals__`。

## 第五步：记录反馈基线
- 正常页面长度
- 触发报错时长度
- 表达式求值时长度
- 是否存在 500、302、403、超时等差异

## 第六步：锁定引擎
- 参考 [[12-常见引擎指纹与差异]]
- 不要只靠一个 payload 下结论。
- 最少要有“求值表现 + 报错信息 + 语法特征”三者中的两项。

## 实战细节
- 如果参数进入 JSON，再被后台渲染，注意引号闭合和转义。
- 如果参数经过多层处理，可能发生二次渲染。
- 如果页面有缓存，别被旧结果误导，必要时加随机值。

## 一套够用的打点顺序
1. `{{7*7}}`
2. `${7*7}`
3. `#{7*7}`
4. `{{1337}}`
5. `{{config}}`
6. `{{request}}`
7. 触发一个可控报错，收集栈信息
