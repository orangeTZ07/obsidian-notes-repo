# SSTI 原理解释

## 一句话定义
`SSTI (Server-Side Template Injection)`：攻击者把可控输入送进服务端模板引擎，在模板渲染阶段被当成表达式、对象访问或指令执行。

## 为什么危险
- 模板引擎通常能读取上下文变量，而上下文里常常带着应用对象、配置对象、请求对象。
- 某些模板引擎支持函数调用、方法访问、过滤器、宏、文件包含，攻击面远大于“字符串替换”。
- 如果能碰到危险对象，SSTI 很容易从“数学表达式求值”升级到“读文件、泄露密钥、命令执行”。

## 你应该建立的核心模型
- 第 1 层：模板分隔符
  - 例如 `{{ ... }}`、`${ ... }}`、`#{ ... }}`、`<%= ... %>`。
  - 这是“语法入口”，不等于一定能执行危险操作。
- 第 2 层：表达式能力
  - 能不能做算术？
  - 能不能读变量？
  - 能不能访问对象属性、索引、方法？
- 第 3 层：上下文对象
  - 有哪些默认对象可以拿来继续扩展？
  - 有没有 `request`、`config`、`self`、`class`、`application` 这类起点？
- 第 4 层：危险能力
  - 能不能访问文件系统？
  - 能不能调用运行时命令？
  - 能不能包含其他模板或触发外带请求？

## 初学者常见误区
- 误区 1：能算 `7*7` 就一定能 RCE。
  - 纠正：这只说明表达式被求值，危险能力取决于引擎能力和上下文对象。
- 误区 2：SSTI 就是 Jinja2。
  - 纠正：Jinja2 只是 CTF 中最常见的一类。PHP、Java、Node 生态都有模板引擎。
- 误区 3：页面不回显就不能打。
  - 纠正：可通过报错、时间差、外带、条件分支来判断。
- 误区 4：模板语法越像代码越危险。
  - 纠正：很多真正危险的利用来自对象链，不是肉眼看上去“像代码”的语法。

## 典型漏洞代码
```python
from flask import request, render_template_string

name = request.args.get("name", "")
return render_template_string("Hello " + name)
```

问题不在 Flask 本身，而在“把用户输入直接拼到模板源码里渲染”。

## SSTI 与其他漏洞的区别
- 与 XSS 的区别
  - XSS 是浏览器执行攻击者脚本。
  - SSTI 是服务端在模板渲染期执行表达式或访问危险对象。
- 与 SQL 注入的区别
  - SQL 注入进入数据库解释器。
  - SSTI 进入模板解释器。
- 与代码注入的关系
  - SSTI 往往不是直接把字符串送给 `eval`，但在高危上下文下最终效果可能接近代码执行。

## CTF 中最常见的利用目标
1. 验证表达式求值：`{{7*7}}`
2. 读取上下文对象：`request`、`config`、`self`
3. 读文件：flag、配置文件、源码片段
4. 命令执行：`id`、`cat /flag`
5. 无回显外带：DNS、HTTP、时间差

## 记忆方法
- 模板注入不是“拼 payload”，而是“借模板引擎上下文做对象探索”。
- 做题顺序通常是：分隔符 -> 表达式 -> 对象 -> 危险对象 -> 结果通道。
